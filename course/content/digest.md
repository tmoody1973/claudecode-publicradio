# Course digest — "Claude Code for Non-Coders (6 Hour Course)" by Nate Herk
Translated for public media professionals. 10 modules, 35 source chapters, 50 station use cases.


## Module 1: Orientation: What Claude Code Actually Is
The mental shift before the first keystroke | ~32 min of source video
SUMMARY: Claude Code has the word 'code' in the name, but you do not write code to use it. It is a chat window that can also open, read, and edit the actual files on your computer — your spreadsheets, your scripts, your show notes — and reach out to things online. This module explains the three Claude products (Chat, Cowork, Code), why Claude Code is the powerful one, and the single analogy that makes the whole thing click: the AI model is an engine, Claude Code is the car built around it, and you are the driver. The rest of the module is mindset: the difference between writing clever prompts and giving the AI real context about your station, why your judgment about what is good still decides everything, and why you should think of yourself as a manager of a very fast, very literal new hire.
WHY IT MATTERS: Most station staff have only ever used AI as a chatbot in a browser tab — retyping the same background every single time, getting generic output back, and concluding it is not worth the trouble. The shift here is that Claude Code can already know your pledge calendar, your style guide, your last twelve underwriting scripts, and your CPB reporting language, because it can read the folders where those things already live. That is the difference between a smart stranger and a colleague who has been at the station for three years.
CONCEPTS:
- Chat vs. Cowork vs. Code [0:01:10]: Anthropic makes three things. Chat is the website you already know. Cowork is a simpler tool for everyday office automation. Claude Code is the most powerful, and despite the name it needs no coding. FOR A STATION: Chat is asking a smart temp a question. Claude Code is that temp with a desk in your building, a key to the file room, and your style guide already read.
- The harness (engine, car, driver) [0:05:09]: The AI model (Opus, Sonnet, Haiku) is the engine. Claude Code is the car built around it — steering, doors, trunk. You are the driver. Swap engines or swap cars; the driver is the constant. FOR A STATION: The model is the transmitter. Claude Code is the studio and console around it. You are still the one deciding what actually goes on the air.
- Access to local files [0:03:15]: The real difference: Claude Code works inside a folder you point it at. It can open, read, and change the files there — and still search the web when it needs to. FOR A STATION: Point it at your 'Fall Drive 2026' folder and it can read last year's scripts, the goal sheet, and the copy that flopped — instead of you re-explaining all of it.
- Context engineering (vs. prompt engineering) [0:16:08]: Prompts are how you ask. Context is what the AI actually knows. As models improve, clever prompt wording matters less — what it knows about your world matters more, and always will. FOR A STATION: A new intern is not useless because you phrased the assignment badly. They are useless until someone tells them what the station does, who's on the desk, and what's on the air tonight.
- Taste and judgment [0:11:55]: As output gets good, the temptation is to stop reading it. The skill that survives is knowing what good looks like — and correcting the system every time it misses, so it misses less. FOR A STATION: AI can draft the newscast tease. You still know whether it sounds like your station or like a press release. Your name is on the air either way.
- Iteration speed and a definition of done [0:19:50]: You almost never get it right in one shot. Build the ugly version fast, see what breaks, feed the correction back. And decide up front what 'done' means, or you will tinker forever. FOR A STATION: Like teaching a kid to ride a bike: hand on the back, correct the lean, let go a little more each run. Pick one number — hours saved, spots logged — and stop when you hit it.
- Vending machine vs. slot machine [0:21:47]: A plain workflow is a vending machine: same input, same output, cheap, never surprises you. An AI agent is a slot machine: powerful when the input is messy and reasoning is needed, but variable, pricier, and it fails in odd ways. FOR A STATION: Emailing the traffic log every Monday needs no AI — that's a vending machine. Reading 200 messy listener emails and sorting what they actually want does.
STATION USE CASES:
- [News] The news director who has never opened a terminal. Situation: You get a 400-page county budget PDF the night before a council vote and one reporter who is already filing two other stories. How Claude helps: Drop the PDF in a folder, point Claude Code at it, and ask it to pull every line item that changed by more than 10% year over year, with page numbers. It reads the whole thing in minutes and cites where it found each figure so your reporter can verify against the source. This is the local-files idea from Chapter 1 — the AI works in your folder, not in a browser tab. Saves: A full evening of skimming becomes ~20 minutes of reading a cited summary GUARDRAIL: Nothing it produces goes on air or online until your reporter has opened the actual PDF page it cited and confirmed the number. AI finds the lead; a human confirms it. Always.
- [Membership & Development] Onboard Claude to your drive the way you'd onboard an intern. Situation: Every fall drive, you rewrite the same pitch scripts from scratch because last year's live in someone's Google Drive and nobody remembers which ones actually converted. How Claude helps: Put your past scripts, your goal sheet, and your notes on what worked and what flopped into one folder — the summer-intern move from Chapter 4. Now Claude Code drafts against your station's actual voice and history instead of generic best practices. Correct it once, tell it why, and the next draft is closer. Saves: Two days of script rewriting becomes an afternoon of editing GUARDRAIL: Scripts and copy only. Never put the donor file in that folder — no names, no addresses, no giving history, no CRM export. The AI needs your language, not your list.
- [Underwriting] Know when the answer is 'this needs no AI at all'. Situation: Your GM wants 'an AI' to handle underwriting spot copy end to end because a vendor demo looked impressive. How Claude helps: Use the vending-machine-vs-slot-machine test from Chapter 5. Emailing the weekly spot schedule is a vending machine — a plain scheduled script, no AI, never fails. Turning a client's messy intake email into a compliant draft spot is genuinely messy input and needs reasoning, so that part earns the AI. Being the person who can draw that line is the point. Saves: Avoids a six-week build for something a five-minute script does GUARDRAIL: FCC rules do not care that a machine wrote it. Every spot still gets human compliance review: no calls to action, no price claims, no qualitative claims. That review never gets automated.
- [Music & Programming] Prep the local-music segment without the tab-switching. Situation: Your local music host spends two hours before every show pulling artist bios, tour dates, and release news across a dozen browser tabs. How Claude helps: This is the 'what your hand reaches for' shift. Instead of opening the same tabs, hand Claude Code the week's playlist file and let it web-search and assemble a single prep sheet per artist. It reads your local file and reaches out online — both at once, which Chat cannot do. Saves: A 2-hour prep block becomes ~25 minutes of review GUARDRAIL: Anything the host says on air about an artist — a tour date, a label, a hometown — gets confirmed against a real source. A hallucinated tour date is your station's error, not the AI's.
- [Leadership] Budget for the dip before you report on it. Situation: You green-light an AI pilot and someone asks for results after two weeks. The team is slower than before and morale is dropping. How Claude helps: That is the dip from Chapter 6, and it is expected, not failure — a short-term productivity cost before the exponential return. Set the definition-of-done up front: one team, one workflow, one metric (hours off show prep, days off the LCSR draft), and a realistic window. Then judge it. Saves: Prevents killing a working pilot three weeks early GUARDRAIL: Disclose to your board and to staff what you are piloting and where. Public media runs on trust; a quiet AI rollout that surfaces later is a much bigger problem than a slow one announced early.
TRY THIS — Give Claude Code a first day at your station. Setup: Once Claude Code is installed (that's the next module), make one folder on your desktop called 'station-context'. Put in only public, non-sensitive things: your style guide, three underwriting scripts you're proud of, last quarter's newsletter, your on-air pledge script. No donor data. Then open Claude Code in that folder.
PROMPT: Read every file in this folder. This is context about the public radio station I work at. Then write me a one-page summary of what you learned, organized as: (1) what this station sounds like — tone, sentence length, what it avoids; (2) three conventions our underwriting copy follows; (3) five things you'd need to know about us that are NOT in these files. For part 3, ask me the questions rather than guessing. Do not write anything for air yet — I just want to know what you actually understand about us.

It is a chat window that can also open the files on your computer — that is the entire difference from the Claude website. | Opening a blank chat every time and re-explaining your station from scratch. That is the single habit this module exists to break. Give it the folder, not the paragraph. | Trusting output because it reads well. The video's own warning: when the drafts get good, people stop reading them. In public media, an unread AI draft that reaches air or a donor is a trust problem, not a typo. | Reaching for an AI agent when a boring scheduled script would do. Agents are slot machines — variable, pricier, they fail in odd ways. Default to the simplest thing that works. | Quitting in the dip. Week one with a new method feels slower than the old way. ## Module 2: First Contact — Point It at a Real Station File
Install it, run it, verify it | ~16 min of source video
SUMMARY: This is your first thirty minutes. You install Claude Code (the desktop app is the easy path — no terminal required), you sign in with a paid Claude plan, and you point it at a folder on your own computer. From there it can find files, read them, and build new ones — the demo in the video takes a pile of raw YouTube stats and returns a finished, formatted Excel workbook in about ten minutes, written entirely in plain English. You then learn the four things that make a prompt work (role, context, what NOT to do, and proof), and how to make the AI verify its own output instead of you catching every mistake. Finally, the money part: what a token is, and when Haiku, Sonnet, or Opus is worth reaching for.
WHY IT MATTERS: Every station already has the file. The pledge-drive results spreadsheet, the music log, the underwriting traffic report, the CPB reporting workbook — they exist, and somebody spends a weekend wrestling each one into a shape a board or a funder can read. This module is the point where that stops being a weekend and starts being a prompt, and where you learn how to prove the numbers are right before anyone sees them.
CONCEPTS:
- Where you run it [0:33:20]: Claude Code runs in the Claude desktop app, in a code editor, or in a terminal window. Same engine underneath. Pick whichever looks least scary. FOR A STATION: Like choosing between the studio board and the remote kit — same signal path, different surface. Start with the desktop app; nothing is lost if you switch later.
- Local file access [0:34:30]: Claude Code can look inside folders on your own machine — find a file you half-remember, read it, edit it, move it, and make new files next to it. FOR A STATION: It is a producer who can actually open the shared drive. 'Find the Q2 pledge results somewhere in Downloads' is a real, working instruction.
- It builds the deliverable, not just the answer [0:36:00]: Ask for analysis and you get a chat reply. Ask for a workbook and you get an actual .xlsx on disk — dashboard tab, per-item scorecard, trends, color-coded. FOR A STATION: The output is the thing you hand the GM or the board, not notes you then have to retype into a spreadsheet at 11pm.
- The four levers of a prompt [0:38:30]: Role (who it is), context and background (what the situation actually is), negative prompting (what NOT to do), and verification (prove it). FOR A STATION: Same as briefing a new intern before a shift: who you are today, what's going on, what not to touch, and show me before it airs.
- Negative prompting [0:40:36]: Explicitly say what not to do. A capable but inexperienced helper will try things you never imagined unless you rule them out up front. FOR A STATION: 'Do not invent a number that isn't in the file. Do not round. Do not include any donor names.' Say it out loud, every time.
- Make the AI prove its work [0:41:10]: Tell it how to check itself. Unverified, a first pass lands around 60% right and you grind it up by hand. Self-verifying, it starts near 80% and you finish in two passes. FOR A STATION: The check the AI runs is the check you would have run: re-add the column, re-count the rows, cross-check the total against the source file. Ask for that check in the prompt.
- Tokens, and picking a model [0:44:10]: You are billed by token — roughly three-quarters of a word. Output tokens cost about five times input. Haiku is fast and cheap, Sonnet is balanced, Opus is the most capable and the most expensive. FOR A STATION: Haiku for sorting and tidying. Sonnet for most real work. Opus when the reasoning has to be right the first time — grant narrative, board analysis, anything a funder reads.
STATION USE CASES:
- [Membership & Development] Turn the pledge-drive export into a board-ready readout. Situation: The drive ended Sunday. You have a 3,000-row export from the CRM and a board meeting Thursday, and nobody has time to build the pivot tables again. How Claude helps: Strip the export down to non-identifying columns first — gift amount, date, hour, premium, new-vs-renewing, channel — and save that anonymized file locally. Point Claude Code at it and ask for a workbook: dashboard tab, hour-by-hour take, new vs. renewing split, comparison to last drive. Add the verification line so it re-sums every column against the source before handing it over. Saves: A two-day spreadsheet weekend becomes about 30 minutes plus a review pass GUARDRAIL: Donor names, emails, addresses, phone numbers and giving history never leave your CRM. Export the numbers, not the people. Strip PII before Claude ever sees the file.
- [Music & Programming] Read the music log without reading the music log. Situation: Your automation system spits out a play log every week. You want to know whether local artist rotation actually hit the target you promised in the last grant, and whether new music share is drifting. How Claude helps: Drop the log export in a folder and ask Claude Code to tally spins by artist, flag which are tagged local, chart the local percentage week over week, and write the counts into a spreadsheet you can reuse. Tell it explicitly not to guess an artist's local status when the tag is blank — flag those rows for a human instead. Saves: A 4-hour manual tally becomes ~20 minutes GUARDRAIL: Claude counts spins; it does not decide licensing or reporting obligations. Anything headed to SoundExchange, ASCAP or BMI goes through the person who actually owns that filing.
- [Grants & Compliance] Assemble the CPB Local Content & Service Report evidence pile. Situation: The LCSR is due and the underlying numbers are scattered across five spreadsheets, three departments and one person's Downloads folder. How Claude helps: Ask Claude Code to find the relevant files across your project folder, read them, and build a single consolidated worksheet with every figure traced back to its source file and cell. Then ask it to prove the work: recompute each total from the raw file and show its arithmetic next to the number it reported. Saves: A 6-hour hunt-and-reconcile pass becomes ~45 minutes GUARDRAIL: Claude assembles and cross-checks; a human signs the report. Never let it fill a gap by estimating — an empty cell stays empty and gets escalated.
- [Digital & Audience] Make the podcast download CSV mean something. Situation: You get a monthly download CSV per show. Leadership asks 'how are the podcasts doing,' and you have never had a clean way to answer with a trend rather than a number. How Claude helps: Point Claude Code at the folder of monthly CSVs. Ask it to merge them, chart downloads per episode and per show over time, flag the episodes that outperformed their show's median, and write it into one workbook you re-run each month. Cheap, structural work — this is a Haiku or Sonnet job, not an Opus one. Saves: A recurring 2-hour monthly chore becomes a 5-minute re-run GUARDRAIL: Download platforms count differently. Ask Claude to state the definition it used for a 'download' and keep the platform's own methodology note attached to the report.
- [Underwriting] Reconcile the traffic report against what actually aired. Situation: A sponsor claims their spots didn't run. You have the traffic log and the as-aired report, and comparing them by eye is how people lose afternoons. How Claude helps: Give Claude Code both exports and ask it to match scheduled spots against as-aired, list every discrepancy with date, time and spot ID, and produce a make-good candidate list. Tell it to show the matching rows for each discrepancy so you can spot-check the logic rather than trust it. Saves: A 3-hour line-by-line reconciliation becomes ~25 minutes GUARDRAIL: Claude finds mismatches; it does not draft the client response or write spot copy. Any copy that reaches air goes through FCC compliance review — no calls to action, no price or qualitative claims.
TRY THIS — Your first real file — a pledge-drive readout, no formulas. Setup: Install the Claude desktop app and sign in with a paid Claude plan. Make a folder on your machine (call it 'station-work'), and put ONE anonymized spreadsheet in it — a pledge-drive export with donor names, emails, addresses and phone numbers deleted before it ever leaves your CRM. Numbers only. Open Claude Code and point it at that folder.
PROMPT: You are a development analyst at a small public radio station. In this folder there is one spreadsheet: our most recent pledge drive results. It has already been stripped of donor names and contact info — it is numbers only, and it must stay that way.  Read the file first and tell me what columns you actually see before you do anything else.  Then build me an Excel workbook called drive-readout.xlsx with: 1. A summary tab: total raised, number of gifts, average gift, median gift, new vs. renewing split. 2. An hour-by-hour tab showing when money actually came in. 3. A premium tab: which thank-you gifts drove the most dollars and which underperformed. 4. A short written 'what this tells us' section a board member could read in two minutes.  DO NOT invent, estimate, or fill in any number that is not in the source file. If a column I asked about doesn't exist, say so and leave it out — do not approximate. DO NOT add any donor names or contact information, even if you find some.  Before you show me the workbook, verify your work: recompute every total and average directly from the source rows, compare it to what you wrote into the workbook, and show me that comparison. If anything doesn't reconcile, fix it and re-check before you tell me you're done.
PITFALLS: It gives you 60% on the first pass unless you ask it to verify. Bake the proof step into the prompt or you become the QA department. | Never point it at a raw CRM or donor export. Strip names, emails, addresses, phones and giving history first — anonymize before the file leaves your system, not after. | 'It made a beautiful spreadsheet' is not 'the numbers are right.' Formatting quality and arithmetic quality are separate things. Spot-check totals against the source yourself the first few times. | Don't reflexively run everything on Opus. Sorting a CSV is a Haiku job; output tokens cost about 5x input, so re-generating a giant workbook repeatedly is where usage disappears. | Model names and prices in the video are a snapshot in time. Check Anthropic's current pricing before you build a budget on them.

## Module 3: Set Up Your Station's Workspace — And Its Guardrails
CLAUDE.md, settings, keys, and what never leaves the building | ~32 min of source video
SUMMARY: This module is where you stop chatting with an AI and start running it inside a workspace you control. You create a project folder, write a CLAUDE.md — a plain-English file the AI reads before it reads anything you type — and a .claude folder that holds your settings. You learn where secrets go (a .env file that never gets shared), what an API key actually is (a password), and how to hand out keys that can only do one narrow job. Then you learn the permission modes that decide how much Claude Code can do without asking you first, and the difference between bypass mode and sleeping well at night. Finally: what actually happens to your data when you hit enter.
WHY IT MATTERS: Everything else in this course is a technique. This is the part that keeps you out of trouble. A station holds donor giving histories, unpublished investigations, personnel files, and CPB reporting data — and the difference between a tool that saves your development director ten hours and a tool that ends her career is entirely the setup you do in this module.
CONCEPTS:
- CLAUDE.md [0:48:20]: A plain-text file in your project that Claude Code reads before it reads your message. It holds your rules, your context, and where things live. FOR A STATION: The station style guide the AI actually reads — every time, without being reminded. Your pronunciation list, your no-go phrases, your beat structure, your donor-data rules.
- Global vs. project CLAUDE.md [0:53:35]: A global file applies to every project on your machine. A project file applies only inside that one folder. You get both, and both get read. FOR A STATION: Global is house style — how the station writes, always. Project is show-level — the rules for pledge drive, or for this one investigation.
- The .claude folder (settings.json) [0:57:43]: A config folder next to your CLAUDE.md. Inside, settings.json lists what Claude Code is allowed to do and what it is flatly denied from doing. FOR A STATION: The door codes. CLAUDE.md is the staff handbook; settings.json is which rooms the badge actually opens. You describe the rules in English and Claude writes the file.
- API keys and the .env file [1:01:11]: An API key is a password that lets one tool pull data from another. It goes in a .env file, which stays on your machine and never gets shared or published. FOR A STATION: The key is a station credit card with your name on it. The .env file is the locked drawer. Never the script, never the email, never the shared folder.
- Permission modes [1:09:59]: How much Claude Code can do before asking. The video shows auto (the default), manual permissions, accept edits, plan mode, and bypass permissions. FOR A STATION: Auto asks before anything consequential. Plan mode makes it show you the rundown first. Bypass is handing an intern the master key and going to lunch.
- Scoped keys / least privilege [1:11:31]: Most tools let you create a key that can only do a few specific things, and only spend a set amount. Give the AI a key that can read but not delete. FOR A STATION: Your CRM key should let the AI read donor counts, never edit or delete a record. If it physically cannot delete, it cannot delete by accident.
- What leaves your machine [1:17:27]: Claude is a closed model. Every message you send goes to Anthropic's servers to be processed. The data is there even when training is turned off. FOR A STATION: Treat the prompt box like a phone call to a vendor, not a note to yourself. Whatever you paste, you have handed to someone outside the station.
STATION USE CASES:
- [Membership & Development] A CRM key that can look but not touch. Situation: Your development director wants Claude Code to summarize lapsed-donor trends by segment, but the last thing anyone wants is an agent writing to the donor database. How Claude helps: Generate a read-only, scoped API key in your CRM — read access only, no edit, no delete, no export-all — and store it in .env. Then add a deny rule in settings.json so Claude Code can never run a delete or write command against it. Ask for aggregate counts, never named records. Saves: A quarterly segment pull that took a week of back-and-forth with IT becomes an afternoon GUARDRAIL: Never paste a donor export, name, address, email, or giving history into the prompt box. Aggregates only. If the key can write, it is the wrong key.
- [News] The newsroom style guide the AI actually follows. Situation: Every reporter re-explains the same rules to the AI: how you refer to the mayor's office, which sources need a second confirmation, that you never use 'slammed' in a lede. How Claude helps: Put all of it in a project CLAUDE.md once. When an output comes back wrong, tell Claude to update its own CLAUDE.md so it never repeats the mistake — the file gets smarter every time you correct it, instead of you retyping the same note. Saves: Ten minutes of setup instructions per session drops to zero GUARDRAIL: Unpublished investigative material and confidential source identities go nowhere near the prompt box. CLAUDE.md should say so explicitly, in writing, at the top.
- [Underwriting] Compliance rules baked into the workspace. Situation: An account executive drafts spot copy and keeps having to strip out calls to action and price claims that would violate FCC underwriting rules. How Claude helps: Write the FCC constraints directly into a project CLAUDE.md — no calls to action, no qualitative or comparative claims, no prices, no inducements to buy. Claude Code reads it before every draft, so the copy comes back closer to compliant on the first pass. Saves: Three compliance rewrites per spot becomes roughly one GUARDRAIL: A CLAUDE.md rule is a first filter, not a lawyer. Every spot still gets a human compliance review before it airs. Never treat the AI's 'this is compliant' as clearance.
- [Leadership] Deciding what the station will and won't send to a vendor. Situation: Staff are already pasting things into AI tools and nobody has written down what is allowed. The board asks what your policy is. How Claude helps: Use this module's plain framing to write one: closed models send every prompt to a vendor's servers; turning off training does not delete the data; scoped keys and settings.json deny-rules are how you enforce limits technically instead of by memo. Put the resulting policy in your global CLAUDE.md so it applies to every project on every machine. Saves: Turns a vague six-month policy debate into a one-page rule everyone can follow this week GUARDRAIL: Donor PII, personnel files, unpublished journalism, legal correspondence, and anything under an NDA. Name them in the policy — 'be careful' is not a policy.
- [Grants & Compliance] Plan mode before it touches your CPB reporting files. Situation: Your Local Content and Service Report data lives in a folder of spreadsheets nobody wants an agent silently rewriting the week before the deadline. How Claude helps: Work in plan mode so Claude Code shows you exactly what it intends to change before it changes anything, and set deny rules in settings.json so it can never delete files in that directory. Then review the plan the way you'd review a rundown. Saves: Removes the 'did it just overwrite my source data' panic entirely GUARDRAIL: Never run bypass permissions mode in a folder containing the only copy of anything. Keep a backup outside the project directory.
TRY THIS — Build your station's CLAUDE.md in ten minutes. Setup: Make a new folder on your desktop — call it 'station-workspace' — and open Claude Code in it. When it asks whether you trust the folder, say yes (you made it). You do not need to know what a JSON file is; you are going to describe what you want in English and let Claude write the file.
PROMPT: I work at a small public radio station. I want to set up this project properly before I do any real work in it.  Please do three things:  1. Create a CLAUDE.md in this folder. Use the example structure I'm about to describe: a short section on who we are, a house style section, a hard 'never do this' section covering donor data and unpublished journalism, and a section on where files live. Ask me questions if you need details about the station — don't guess.  2. Create a .claude folder with a settings.json. I want you to be explicitly DENIED from deleting any file in this project, and denied from running any command that removes files. I don't know the syntax — write it for me and then explain in plain English what each rule does.  3. Create a .env file with a placeholder for an API key, and make sure it can never be committed or shared. Explain to me in two sentences why the key goes there and not in CLAUDE.md.  When you're done, show me all three files and walk me through them like I've never seen a config file before — because I haven't.
PITFALLS: Bypass permissions mode is exactly what the name says — Claude Code never stops to ask. The video's own team had an agent email a discount code to 150,000 people, and is honest about the real cause: not the permission mode, but an agent with access to too many tools. Scoped keys are your first line of defense; permission modes are the second. ## Module 10: What This Actually Costs — Tokens, Caching, and Session Limits
The answer when your GM asks the budget question | ~32 min of source video
SUMMARY: Claude charges you by the token — roughly one token per word — and every time you send a message it rereads the whole conversation from the beginning. That means cost compounds instead of adding: message one might be 500 tokens, message thirty might be 15,500. Prompt caching is what stops that from bankrupting you: Claude stores the unchanged part of your session and re-reads it at 10% of normal input cost. This module teaches what gets cached (system instructions, your CLAUDE.md, the conversation so far), how long the cache lives, and — most usefully — the everyday moves that silently destroy it and make your next message cost full freight.
WHY IT MATTERS: Public media runs on thin margins, and somebody in leadership will ask what an AI tool costs the station before anyone is allowed to use it. The honest answer is that the plan price is fixed but the value you get out of it swings enormously with habits — one person who understands caching gets several times the work out of the same subscription as someone who doesn't. As the video puts it: most people don't need a bigger plan, they have a context hygiene problem.
CONCEPTS:
- Token [5:28:40]: The smallest unit of text an AI model reads and charges you for. Roughly one token per word — not exactly, but close enough to budget with. - Compounding cost [5:28:50]: Every time you send a message, Claude rereads the entire conversation from the start. So costs multiply as a chat grows — they don't just add up. ## Glossary
- .env: File holding secrets like API keys. Kept local and excluded from anything you publish or share.
- .gitignore: A list of files GitHub should never upload — API keys, passwords, anything sensitive. Check it before your first push.
- /clear: Wipes the session clean. Fresh context, zero memory of what you just did — so hand off first.
- /compact: Claude Code command that summarizes the conversation so far and keeps going. Runs automatically when you near the limit — often too late.
- /context: A command showing exactly what is eating your tokens right now — history, MCP overhead, loaded files.
- /cost: A command showing actual token usage and estimated spend for the current session.
- Agent: AI that reasons and decides in a loop rather than following fixed steps. Powerful, variable, more expensive, fails in surprising ways.
- Agent team: Several agents that share a task list and can talk to each other, debate, and reach consensus. More expensive than sub-agents. Experimental feature.
- API key: A password letting one tool pull data from another. Anyone holding it can use your account.
- Auto-memory: A Claude Code setting where it writes and updates a memory file on its own as you work. Toggle it with /memory.
- Backlink: A link from one note to another. Follow it to jump from 'spring drive 2024' to 'sustainer messaging.'
- Brand assets folder: A folder with your logo and brand guidelines that you point Claude at, so what it builds comes out in your colors and type.
- Bypass permissions: Mode where Claude Code never asks for approval. Also called 'dangerously skip permissions.' It means it.
- Cache create: Writing content into the cache for the first time. A one-time cost that pays off on the next turn.
- Cache read: Reusing already-cached content. Costs about 10% of normal input tokens — roughly ten times cheaper.
- CLAUDE.md: Plain-English instructions file Claude Code reads before every message. Your standing rules and context.
- CLI: Command-line interface. A tool you operate by typing commands instead of clicking. Free, fast, and easy for Claude to drive.
- Connector: A prebuilt link between Claude and a tool you already use — Slack, Gmail, Google Drive. You switch it on in settings.
- Context: Everything the AI actually knows about your situation: your files, your history, your priorities. Not the same as your prompt.
- Context engineering: Filling the AI's working memory with exactly the right information. Karpathy's phrase. The durable skill.
- Context rot: The quality drop that happens as a session's context window fills up. Also called 'the dumb zone.'
- Context window: How much the AI can hold in its head at once — like a producer's prep folder for one show. It is finite.
- Cron: The old term for 'run this on a clock.' You don't need to learn the syntax; you can just say 'weekdays at 6am.'
- Deploy: Moving your work from a private preview on your own computer to a public address the world can open.
- Dynamic workflow: Claude fans a big job out to many sub-agents at once — sometimes dozens. Powerful, but it eats your usage limit fast.
- Environment variable: A password or API key handed to a program at startup instead of written into its files. Keeps secrets out of shared code.
- Evals: A set of known inputs with known correct outputs. Run them against your automation to see how often it gets it right before you trust it.
- Front matter: The settings block at the top of a sub-agent's markdown file: name, description, allowed tools, model, memory. Everything below it is the instructions.
- GitHub: A cloud home for your project's files that keeps every past version, so any change can be traced or undone.
- Haiku / Sonnet / Opus: Claude's models: fast-and-cheap, balanced, and most-capable-and-priciest. Same tool, different horsepower.
- Harness: The tool wrapped around an AI model that gives it hands — file access, web search, memory. Claude Code is a harness.
- Index file: A table of contents the AI reads first, so it knows which note to open instead of reading everything.
- Input vs output tokens: What you feed the model vs what it writes back. Output costs about 5x more per token.
- Knowledge graph: Storage that records how things relate — 'Jordan works at Acme,' 'Acme sponsors Morning Edition' — not just that they're linked.
- Lint / health check: Periodically asking the AI to audit its own notes for gaps, contradictions, and missing information.
- Local host: A private preview of your site that runs only on your own machine. Nobody else can see it.
- Markdown: Plain text with light formatting. Opens in any editor, readable in 20 years, needs no software to survive.
- MCP: Model Context Protocol. A standard way to plug Claude Code into another tool, like Gmail or QuickBooks.
- MEMORY.md: The index file that points to those individual memory files. One fact per file; MEMORY.md is the table of contents.
- Misfire: A sub-agent fires when you didn't want it, or stays silent when you did. Fixed by rewriting its description, not its instructions.
- Modal / trigger.dev: Cheap hosting services that run a small script on a schedule or on a webhook. Pennies per run. Used in the video's build.
- Model: The raw AI brain — Opus, Sonnet, Haiku, Fable. Interchangeable. Claude Code runs on the same ones as Claude Chat.
- Negative prompt: The 'do NOT do this' part of your instructions. Guardrails, stated before the work starts.
- North star metric: The one number you pick before you build, that tells you afterward whether the build was worth it.
- OAuth: The 'Sign in with Google, allow access' step. How you grant Claude reach into your own Workspace account, revocably.
- Obsidian: A free app that displays your markdown files and draws the links between them as a picture. Optional — the files work without it.
- Orchestrator: Your main session. The only one you talk to. It hands work out to sub-agents and relays their findings back to you.
- Peak hours: 8am-2pm Eastern on weekdays, when the five-hour session window drains faster because demand is high.
- Permission mode: Setting controlling how often Claude Code stops to ask before acting. Auto is the default.
- Prefix matching: How the cache works: it reuses content only if the front of the request is identical to last time.
- Progressive disclosure: Claude skims every Skill's description, then fully reads only the one it needs. Keeps forty Skills from crowding out your actual work.
- Prompt engineering: Wording a request well. Was a big deal; matters less each year as models improve on their own.
- Repository (repo): One project's folder on GitHub — all the files plus the full history of who changed what and when.
- Routine: A saved prompt Claude Code runs by itself on a schedule, on Anthropic's servers, with your machine off.
- Run history / logs: The record of every automated run. When one fails, you paste the failed run back to Claude and ask it why.
- Scheduled task (local): Same idea, but it runs on your own computer — so your machine has to be on and awake.
- Scoped key: An API key restricted to specific capabilities and spending limits. Read-only, or one narrow function.
- Semantic search: Finding notes by meaning rather than exact wording. Searching 'listener complaints' surfaces a note that says 'audience feedback.'
- Session limit: On a Claude subscription you get a rolling 5-hour usage window plus a weekly cap, instead of per-token billing.
- settings.json: Config file in the .claude folder listing what Claude Code is allowed and denied from doing.
- Skill: A folder with a SKILL.md file describing one procedure. Claude loads it on its own when the request matches.
- Sub-agent: A separate AI worker with a fresh, empty memory, given one job by your main session. Reports back a summary. Defined in one markdown file.
- Theory of constraints: Work only compounds at the bottleneck. Unclog the biggest one, and the next one becomes the biggest.
- Token: The smallest chunk of text an AI reads and bills for. Roughly one word, give or take.
- Tool-agnostic: Set up so your work survives switching AI apps. Files, folders, and your own keys move; app-specific connectors don't.
- TTL: Time to live. How long cached content stays valid: one hour on a subscription, five minutes on the API.
- Vercel: A service that takes the files in your GitHub repo and publishes them as a live website at a real web address.
- Verification: Making the AI check its own output against something real before it hands the work back to you.
- Webhook: A doorbell. One system pings another the instant something happens, instead of the second system constantly checking.
- Workflow: A fixed if-this-then-that sequence with no AI in it. Predictable, cheap, boring, almost never breaks. Often the right answer.
- YAML front matter: The few labeled lines at the top of a SKILL.md — name, description, when to use it. Claude reads these to decide.
